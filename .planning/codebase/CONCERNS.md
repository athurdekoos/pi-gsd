# Codebase Concerns

**Analysis Date:** 2026-03-05

## Tech Debt

**CLI tooling has zero test coverage:**
- Issue: All 11 library modules in `gsd/bin/lib/*.cjs` (~5400 lines) have no dedicated unit or integration tests
- Files: `gsd/bin/lib/state.cjs`, `gsd/bin/lib/phase.cjs`, `gsd/bin/lib/verify.cjs`, `gsd/bin/lib/frontmatter.cjs`, `gsd/bin/lib/roadmap.cjs`, `gsd/bin/lib/init.cjs`, `gsd/bin/lib/commands.cjs`, `gsd/bin/lib/config.cjs`, `gsd/bin/lib/milestone.cjs`, `gsd/bin/lib/template.cjs`, `gsd/bin/lib/core.cjs`
- Impact: State management, phase lifecycle, roadmap parsing, and verification logic can silently break — documented in `docs/ops/testing.md` as the "major gap"
- Fix approach: Add test suites for each module, starting with `core.cjs` (shared foundation) and `state.cjs` (most complex state management)

**Custom YAML frontmatter parser instead of standard library (ADR-001):**
- Issue: Hand-rolled YAML parser in `gsd/bin/lib/frontmatter.cjs` (299 lines) handles only a subset of YAML
- Files: `gsd/bin/lib/frontmatter.cjs`
- Why: Zero-dependency constraint (ADR-004) prevents using `js-yaml` or similar
- Impact: Edge cases in nested objects, multi-line values, or special characters may parse incorrectly; maintenance burden for parser bugs
- Fix approach: Accept as intentional trade-off per ADR-001/ADR-004, but ensure comprehensive unit tests exist for the parser (currently partially tested in `tests/unit-frontmatter.test.ts`)

**Duplicated test harness functions across test files:**
- Issue: `testSync()` and `testAsync()` helper functions are copy-pasted into every test file instead of imported from a shared location
- Files: All 16 test files in `tests/*.test.ts`
- Impact: Any change to test output format requires editing every file; inconsistency risk
- Fix approach: Extract `testSync`/`testAsync` into `tests/harness/test-helpers.ts` and import across all suites

**No package-lock.json:**
- Issue: `package-lock.json` not present despite using npm
- Files: Project root
- Impact: Non-deterministic dependency resolution; different developers may get different versions of `tsx` and `typescript`
- Fix approach: Run `npm install` to generate lockfile and commit it

## Security Considerations

**GSD_HOME set via process.env in constructor:**
- Risk: `GsdPathResolver` constructor sets `process.env.GSD_HOME` as a side effect, which persists for the entire Node.js process
- Files: `extensions/gsd/path-resolver.ts` (constructor)
- Current mitigation: Only one GsdPathResolver instance created per Pi session
- Recommendations: Document the side effect; consider making GSD_HOME a read-only property rather than env var mutation

**git operations use execSync without input sanitization:**
- Risk: Phase names or commit messages containing shell metacharacters could cause unexpected behavior in `execSync` calls
- Files: `gsd/bin/lib/commands.cjs` (commit function), `gsd/bin/lib/phase.cjs` (branch operations)
- Current mitigation: Phase names go through `generateSlugInternal()` which strips special chars; commit messages are typically generated by the LLM
- Recommendations: Audit all `execSync` calls for proper argument escaping; use `execFileSync` where possible to avoid shell interpretation

**Forbidden files list is advisory only:**
- Risk: The codebase mapper agent's `<forbidden_files>` block relies on LLM compliance; no programmatic enforcement prevents reading `.env` or credential files
- Files: `agents/gsd-codebase-mapper.md` (forbidden_files section)
- Current mitigation: LLM instruction compliance; `.env` files typically gitignored
- Recommendations: Consider adding file-pattern checks in `gsd-tools.cjs` to refuse reading files matching secret patterns

## Performance Bottlenecks

**CLI init commands assemble large JSON payloads:**
- Problem: `gsd-tools.cjs init execute-phase` and similar compound commands read multiple files and build large JSON responses
- Files: `gsd/bin/lib/init.cjs` (710 lines, 12+ init functions)
- Measurement: Not benchmarked, but large payloads (>50KB) trigger temp-file fallback in `core.cjs`
- Cause: Single-call pattern assembles all possible context upfront
- Improvement path: Acceptable trade-off — one large call is faster than many small sequential calls from the LLM

**Test runner executes suites serially:**
- Problem: `tests/run-all.ts` runs all 16 suites sequentially via `execSync`
- Files: `tests/run-all.ts`
- Measurement: Total runtime depends on suite count; E2E tests add significant time
- Cause: Simplicity — serial execution avoids output interleaving
- Improvement path: Parallel execution with worker threads or concurrent child processes; output buffering per suite

## Fragile Areas

**Path resolver — single point of failure:**
- Files: `extensions/gsd/path-resolver.ts`
- Why fragile: If any of the 4 rewrite rules break, ALL commands fail — no command can execute without correct path resolution
- Common failures: Regex patterns not matching new path formats; `__dirname` resolution changing with module system changes
- Safe modification: Run full test suite (`npx tsx tests/run-all.ts`) after any change; unit tests cover all 4 rules and edge cases
- Test coverage: Good — `tests/unit-path-rewrite.test.ts` (162 lines), `tests/unit-exec-context.test.ts` (185 lines)

**STATE.md dual representation:**
- Files: `gsd/bin/lib/state.cjs` (732 lines)
- Why fragile: STATE.md has YAML frontmatter (machine-readable) AND markdown body (human-readable) that must be kept in sync
- Common failures: Frontmatter updates not reflected in body; body edits breaking frontmatter structure
- Safe modification: Always use `gsd-tools.cjs state update/patch` — never edit STATE.md manually
- Test coverage: None — `state.cjs` has no dedicated tests

**Phase renumbering logic:**
- Files: `gsd/bin/lib/phase.cjs` (901 lines)
- Why fragile: Phase insert/remove operations renumber all subsequent phases, updating both directory names and ROADMAP.md references
- Common failures: Renumbering may miss references in STATE.md, plan files, or cross-phase dependencies
- Safe modification: Always use `gsd-tools.cjs phase add/insert/remove` — validate with `gsd-tools.cjs validate consistency` after
- Test coverage: None — `phase.cjs` has no dedicated tests

**Frontmatter reconstruction:**
- Files: `gsd/bin/lib/frontmatter.cjs` (299 lines)
- Why fragile: Custom YAML serializer must produce valid YAML for diverse value types (arrays, nested objects, booleans, numbers, multi-line strings)
- Common failures: Edge cases in special characters, empty arrays, nested objects with mixed types
- Safe modification: Add test cases for any new frontmatter structures before using them
- Test coverage: Partial — `tests/unit-frontmatter.test.ts` covers basic cases

## Scaling Limits

**Markdown-as-code approach:**
- Current capacity: Works well for 30 commands, 30 workflows, 11 agents
- Limit: LLM context windows constrain individual workflow size; largest workflow (`new-project.md`) is 1116 lines
- Symptoms at limit: Workflows that exceed context length will be truncated; LLM may lose track of steps
- Scaling path: Split large workflows into smaller sub-workflows; use agent delegation for complex steps

**gsd-tools.cjs single-process model:**
- Current capacity: All CLI operations are synchronous and single-threaded
- Limit: Very large `.planning/` directories with hundreds of phases could slow file scanning
- Symptoms at limit: Noticeable delay on init commands that scan all phase directories
- Scaling path: Add caching or indexing for phase directory lookups

## Dependencies at Risk

**tsx (devDependency):**
- Risk: Actively maintained but used only for test execution; breaking changes could prevent test runs
- Impact: Cannot run tests without tsx
- Migration plan: Switch to `node --loader ts-node/esm` or native Node.js TypeScript support (available in Node 22+)

**Pi Extension SDK (peer dependency):**
- Risk: Pi SDK API changes could break event subscriptions or command registration
- Impact: Extension fails to load — all GSD functionality lost
- Migration plan: Compliance tests (`tests/compliance.test.ts`) validate SDK contract; update extension layer when API changes

## Missing Critical Features

**No lockfile:**
- Problem: No `package-lock.json` present
- Blocks: Reproducible builds and consistent dependency resolution across environments

**No CI/CD pipeline:**
- Problem: No GitHub Actions or CI configuration
- Current workaround: Tests run manually via `npx tsx tests/run-all.ts`
- Blocks: Automated regression detection, PR quality gates

## Test Coverage Gaps

**CLI tooling (highest priority):**
- What's not tested: All 80+ commands in `gsd/bin/lib/*.cjs` — state management, phase lifecycle, roadmap parsing, verification, frontmatter CRUD, template filling, milestone operations
- Files: `gsd/bin/lib/state.cjs`, `gsd/bin/lib/phase.cjs`, `gsd/bin/lib/verify.cjs`, `gsd/bin/lib/init.cjs`, `gsd/bin/lib/commands.cjs`, `gsd/bin/lib/roadmap.cjs`, `gsd/bin/lib/milestone.cjs`, `gsd/bin/lib/template.cjs`, `gsd/bin/lib/config.cjs`
- Risk: Core state operations (phase add/remove/renumber, state progression, roadmap updates) can break silently
- Priority: High
- Difficulty to test: Moderate — functions take `cwd` parameter, testable with temp directories

**Workflow behavior (medium priority):**
- What's not tested: Markdown workflow files only tested indirectly via integration tests that verify command output transformations
- Files: `gsd/workflows/*.md` (30+ files)
- Risk: Workflow instructions may reference nonexistent gsd-tools commands or templates
- Priority: Medium
- Difficulty to test: Hard — workflows are LLM-interpreted; would need end-to-end testing with actual LLM

**Hot-reload edge cases:**
- What's not tested: Command `.md` file deleted or renamed between registration and invocation
- Files: `extensions/gsd/commands.ts` (handler re-reads file at invocation time)
- Risk: Runtime error if command file removed after Pi startup
- Priority: Low
- Difficulty to test: Easy — delete file, invoke handler, check error handling
